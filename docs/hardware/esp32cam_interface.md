# ESP32-CAM Interface — SecurityNode

## Overview

The **ESP32-CAM** module is connected to the SecurityNode main board via a dedicated header (J3). It serves as the auxiliary vision-processing module, capturing images and performing face-coverage analysis upon request from the ESP32-C3 controller.

> **Stage:** SecurityNode is a **design-stage / not-yet-fabricated** project. The hardware facts below (J3 pinout, power gating, U4/U5 roles, programming path) are verified against the schematic netlist (`assets/SecurityNode.pdf`). The UART message flow and vision workflow are a **proposed firmware design** — no firmware exists yet — and are labeled as such.

## Physical Connection — J3 Header

The ESP32-CAM module (commonly the AI-Thinker variant with OV2640 camera) mounts onto the board through header J3. Only the nets actually used by SecurityNode are broken out: power, ground, the UART1 link, and the CAM boot strap. The SD-card / PSRAM GPIO breakouts are *not* routed on this board.

### Pinout (J3 — from the schematic netlist)

| J3 Pin | Net | Function | Connected To |
|--------|-----|----------|--------------|
| P$2 | CAM_TXD_RAW | CAM UART TX (CAM U0TXD/GPIO1 → C3) | ESP32-C3 **IO0 (UART1_RX)**, via U5 + R20 1k |
| P$3 | CAM_RXD_RAW | CAM UART RX (CAM U0RXD/GPIO3 ← C3) | ESP32-C3 **IO1 (UART1_TX)**, via U5 + R21 1k |
| P$6 | B_GPIO0_CAM | CAM boot / GPIO0 strap | J6 2-pin boot jumper / J7 pin4 (not driven by the C3) |
| P$8 | CAM_3V3 | CAM 3.3V (the CAM module's own regulator output) | Local to CAM; also the **U5 control input** (presence gate) |
| P$9 | CAM_5V | Gated 5V supply to the CAM | **U4 (TPS2553) output** (U4 IN = 5V rail) |

> GND pins are present on the connector and tie to the global ground plane.

> **Note:** CAM_3V3 is the ESP32-CAM module's *own* internal regulator output, not the main-board AP2112K (U1) 3.3V rail. The C3↔CAM UART nets pass through the U5 isolation switch (see below). Cross-reference any change against `assets/SecurityNode.pdf` (schematic = ground truth).

## Power Supply

The ESP32-CAM is supplied **only with gated 5V** from the main board (`CAM_5V`). Its 3.3V logic/sensor supply (`CAM_3V3`) is generated by the ESP32-CAM module's **own on-board regulator** — the main-board AP2112K (U1) does *not* feed the CAM:

| Rail | Source | Purpose |
|------|--------|---------|
| CAM_5V | **TPS2553 (U4)** gated output (U4 IN = 5V rail) | Sole power input to the ESP32-CAM module (its internal 5V→3.3V regulator and RF) |
| CAM_3V3 | **ESP32-CAM module's own internal regulator** | I/O logic, camera sensor digital supply; also drives the U5 presence-gate control (P$8) |

### Power Considerations

- The ESP32-CAM can draw up to **500mA peak** during Wi-Fi transmission + camera operation.
- The 5V to the CAM is **gated and current-limited by U4 (TPS2553)**: enabled by the C3 via IO7/CAM_EN (→ R9 100R → U4 EN), current limit set by R13 (21k5), with FAULT reported back to the C3 on IO3/CAM_FAULT. U4 is held **OFF by default** during C3 boot by R11 (100k pulldown). Ensure the R13 current-limit threshold is above the CAM's peak draw.
- The main-board AP2112K (U1) 3.3V rail powers **only the C3 (U3) and U5** — it is not loaded by the CAM, so CAM current does not affect the 3.3V budget.
- Place decoupling capacitors close to J3 for the CAM_5V input; the CAM_3V3 decoupling lives on the CAM module itself.

## UART Communication

A dedicated UART channel connects the ESP32-C3 and the ESP32-CAM over **UART1**. It is independent of the C3's own programming/console path (the C3 is flashed over native USB on IO18/IO19, with UART0 on GPIO20/21 broken out to the J1 debug header). The CAM link physically passes through the U5 isolation switch (see below); it does **not** share any wire with the USB or UART0 path.

| Parameter | Value |
|-----------|-------|
| UART Port | ESP32-C3 UART1 — **IO0 = RX, IO1 = TX** (via U5 + R20/R21 1k) |
| Baud Rate | 115200 (default) |
| Data Bits | 8 |
| Parity | None |
| Stop Bits | 1 |
| Flow Control | None |

> **Direction note:** On the C3, **IO0 is the RX** (receives the CAM's TX, CAM_TXD_RAW) and **IO1 is the TX** (drives the CAM's RX, CAM_RXD_RAW). GPIO8 and GPIO9 are *not* on this link — they are STRAP_GPIO8 and the BOOT button (SW1) respectively.

### Message Flow *(proposed firmware design — not yet implemented)*

```
ESP32-C3                      ESP32-CAM
   |                               |
   |--- CMD_CAPTURE --------------->|
   |                               |
   |<-- ACK_CAPTURE ---------------|
   |                               |
   |                               | [Capture image]
   |                               | [Run face analysis]
   |                               |
   |<-- RESULT (CLEAR/SUSPICIOUS) --|
   |                               |
```

Detailed command structure is documented in `docs/firmware/cam_protocol.md`.

## Boot and Programming

The ESP32-CAM is **not** programmed through the SecurityNode board's USB connector. There is no on-board USB path to the CAM, and U5 is **not** a programming multiplexer (see the U5 section below). The CAM is flashed out-of-band:

### Programming via J7 + External USB-UART Adapter

- Header **J7** (6-pin) breaks out the CAM's raw programming nets: `[1=GND, 2=CAM_TXD_RAW, 3=CAM_RXD_RAW, 4=B_GPIO0_CAM, 5=CAM_3V3, 6=CAM_5V]`.
- Connect an **external USB-to-UART adapter** (e.g. an FTDI/CP210x dongle) to J7: adapter RX ↔ CAM_TXD_RAW, adapter TX ↔ CAM_RXD_RAW, common GND.
- The CAM's GPIO0 boot strap (`B_GPIO0_CAM`) is asserted by the **J6 2-pin jumper** (also exposed on J7 pin4 + R17 10k) — *not* by any ESP32-C3 GPIO.

### Entering ESP32-CAM Bootloader Mode

1. Connect the external USB-UART adapter to J7 (TXD/RXD/GND).
2. Hold CAM GPIO0 LOW by fitting the **J6 boot jumper** (B_GPIO0_CAM).
3. Power-cycle / reset the ESP32-CAM.
4. The ESP32-CAM enters UART download mode; flash with `esptool`/IDE.
5. Remove the J6 jumper and reset for normal (run) operation.

> The ESP32-C3 *could* in firmware act as an application-level UART relay over the UART1 link for remote/in-field updates, but that is a proposed/design-stage feature and is not a hardware programming path — the as-built flashing method is J7 + external adapter.

## Analog Switch — SN74LVC2G66DCTR (U5)

### Function

U5 is a 2-channel **analog switch used as an automatic, CAM-presence-gated series isolation switch** on the C3↔CAM **UART1** link. Each of the two channels sits in series with one UART line (CAM_TXD/RXD_RAW ↔ the C3-side _ISO nets), connecting the C3 to the CAM only when a powered CAM module is seated.

It is **not** a USB-to-UART programming multiplexer, it is **not** host- or GPIO-selectable, and it does **not** touch the USB or UART0 path.

| U5 control (1C / 2C) tied to | UART1 link state | Condition |
|------------------------------|------------------|-----------|
| CAM_3V3 present (≈ HIGH) | **Connected** | A powered ESP32-CAM is seated (its own 3.3V is up) |
| CAM_3V3 absent (≈ LOW) | **Isolated** | CAM not installed or unpowered |

### Control Logic

- U5's control pins (1C / 2C) are **hard-tied to CAM_3V3** — the ESP32-CAM module's own 3.3V regulator output (J3 P$8). There is no microcontroller or jumper control of U5.
- This makes the UART link **auto-connect** when a powered CAM is present and **auto-disconnect** when the CAM is absent or off, isolating the C3's UART1 pins from a floating/dead connector.
- The switch has low on-state resistance (~few Ω), so it does not meaningfully degrade the signal at 115200 baud.

## Vision Processing Workflow *(proposed firmware design — not yet implemented)*

1. **Trigger**: Door open or PIR motion event detected by ESP32-C3.
2. **Command**: ESP32-C3 sends `CMD_CAPTURE` to ESP32-CAM via UART.
3. **Capture**: ESP32-CAM captures a frame from the OV2640 camera.
4. **Analysis**: ESP32-CAM runs face detection / coverage algorithm (e.g., Haar cascade, lightweight CNN, or simple skin-tone analysis).
5. **Result**: ESP32-CAM returns one of:
   - `RESULT_CLEAR` — face is visible, no threat detected.
   - `RESULT_SUSPICIOUS` — face is covered (mask, helmet, balaclava), threat detected.
   - `RESULT_ERROR` — capture or analysis failed.
6. **Action**: ESP32-C3 evaluates result and activates alarm if necessary.

## Module Isolation

The modular architecture allows the system to operate without the ESP32-CAM:

| Scenario | ESP32-CAM | ESP32-C3 Behavior |
|----------|-----------|-----------------|
| CAM connected and responsive | Active | Full vision-assisted alarm mode |
| CAM connected but not responding | Unresponsive | Timeout after 5s → fallback to basic alarm |
| CAM not installed | Absent | Basic alarm mode (sensor → immediate alarm) |

## Design Notes

- The ESP32-CAM AI-Thinker module includes an on-board PCB antenna; ensure no copper pours or traces block the antenna area on the main PCB.
- The camera ribbon cable (OV2640) connects to the ESP32-CAM module, not the main board. Ensure adequate clearance for the camera module.
- Keep the UART1 traces between the ESP32-C3, U5, and J3 short and matched in length to minimize skew.
- The ESP32-CAM supplies its own 3.3V (CAM_3V3) from its on-board regulator; the main-board AP2112K (U1) 3.3V rail is not loaded by the CAM, so no extra 3.3V regulator is needed for the CAM module.
- The AI-Thinker ESP32-CAM does not have an on-board USB-to-UART bridge; therefore external programming is required, done via header **J7** with an external USB-UART (FTDI/CP210x) adapter plus the J6 boot jumper. U5 is an automatic UART isolation switch and is **not** a programming path.
